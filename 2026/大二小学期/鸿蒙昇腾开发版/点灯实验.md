# 点灯实验一：交替点亮

## 代码
最后正确的代码
``` C
#include <unistd.h>
#include "stdio.h"
#include "ohos_init.h"
#include "cmsis_os2.h"
#include "iot_gpio.h"
#define LED_TEST_GPIO 9 // for hispark_pegasus
void *LedTask(const char *arg)
{
    // 初始化GPIO
    IoTGpioInit(LED_TEST_GPIO);
    // 设置为输出
    IoTGpioSetDir(LED_TEST_GPIO, IOT_GPIO_DIR_OUT);
    (void)arg;
    while (1)
    {
        // 输出低电平 (0)
        IoTGpioSetOutputVal(LED_TEST_GPIO, 0);
        // osDelay 的单位是系统 Tick。假设1个Tick是10ms，30代表延时300ms
        // 延时期间会释放CPU，允许其他任务运行
        osDelay(30);
        // 输出高电平 (1)
        IoTGpioSetOutputVal(LED_TEST_GPIO, 1);
        osDelay(30);
    }
    return NULL;
}
void led_demo(void)

{

    osThreadAttr_t attr;
    attr.name = "LedTask";
    attr.attr_bits = 0U;
    attr.cb_mem = NULL;
    attr.cb_size = 0U;
    attr.stack_mem = NULL;
    attr.stack_size = 1024;             // 🟢 修复：扩大栈空间，防止 printf 导致栈溢出
    attr.priority = osPriorityNormal;   // 🟢 修复：使用标准的系统正常优先级 
    if (osThreadNew((osThreadFunc_t)LedTask, NULL, &attr) == NULL) {
        printf("[LedExample] Failed to create LedTask!\n"); // 🟢 修复：拼写错误
    }
}
// 🟢 修复：关键！必须注册该函数，系统启动时才会去调用 led_demo 从而创建点灯线程

APP_FEATURE_INIT(led_demo);

```

以下是对本次实验的系统性总结、问题复盘以及解决方案：

## 一、 实验核心逻辑

一个标准的 OpenHarmony 驱动开发（以点灯为例）包含三个层次：

1. **构建层（GN/Ninja）**：通过 `BUILD.gn` 声明源文件、头文件路径（`include_dirs`），并将其打包为静态库（`static_library`）和组件（`lite_component`）。
    
2. **业务层（C 代码）**：
    
    - 调用 `IoTGpioInit` 和 `IoTGpioSetDir` 初始化硬件引脚。
        
    - 使用 `osThreadNew` 创建独立的任务线程（`LedTask`），在 `while(1)` 循环中控制引脚高低电平切换。
        
3. **系统层（启动绑定）**：使用 `APP_FEATURE_INIT` 宏将入口函数注册到系统的自动启动段中。
    

## 二、 实验过程中遇到的问题及解决方案

在本次实验中，我们共同诊断并修复了以下 5 个关键问题：

### 1. 代码拼写与接口误用（逻辑错误）

- **【现象】**：在原始代码中，`while(1)` 循环内误用了设置引脚方向的函数 `IoTGpioSetDir(LED_TEST_GPIO, 0)`。
    
- **【原因】**：手误或对 API 不熟悉。`IoTGpioSetDir` 是用来配置引脚是输入（IN）还是输出（OUT）的。
    
- **【解决】**：修改为控制输出电平的正确接口：**`IoTGpioSetOutputVal(LED_TEST_GPIO, 值)`**（0 为低电平，1 为高电平）。
    

### 2. VS Code 提示 #include 错误（编辑器环境问题）

- **【现象】**：代码中 `#include "ohos_init.h"` 等头文件下方出现红色波浪线，提示“检测到 #include 错误，更新 includePath”。
    
- **【原因】**：代码本身没问题（可在 Linux 下正常编译），但 Windows 端的 VS Code 找不到 OpenHarmony 源码中这些头文件的绝对路径。
    
- **【解决】**：
    
    - 修改 `.vscode/c_cpp_properties.json`，在 `"includePath"` 中手动添加头文件所在的绝对路径（如 `D:/3.0-lts/src/...`）。
        
    - 或是通过编译生成 `compile_commands.json` 并导入 VS Code 自动配置。
        

### 3. 缺少启动注册宏（运行无响应）

- **【现象】**：编译通过且烧录成功，但板子上的 LED 灯毫无反应，串口也无提示。
    
- **【原因】**：在嵌入式 OS 中，`main` 函数被内核占用。如果仅仅定义了 `led_demo` 函数，内核在启动时根本不知道它的存在。
    
- **【解决】**：在代码最后一行加上 **`APP_FEATURE_INIT(led_demo);`**。将其注册到系统的应用启动序列中，使其在系统就绪后被自动调用。
    

### 4. 线程延时函数误用导致系统卡死（RTOS 调度机制问题）

- **【现象】**：使用高优先级（`26`）创建线程，并在循环内使用 `usleep(300000)`。在某些 RTOS 底层可能导致系统高频复位或死机。
    
- **【原因】**：`usleep` 属于忙等待（死等），高优先级的 `LedTask` 会死死霸占 CPU。这导致排在它后面的系统低优先级任务（如网络、看门狗喂狗任务）无法执行，最终**触发看门狗硬件复位**。
    
- **【解决】**：将 `usleep(300000)` 替换为标准的 CMSIS-RTOS V2 延时函数 **`osDelay(30);`**。它在延时期间会**主动释放 CPU 资源**，让其他任务轮流运行。
    

### 5. 线程栈空间不足（潜在死机隐患）

- **【现象】**：原本设置 `stack_size = 512` 字节，后续若加入 `printf` 打印调试信息，系统极易崩溃。
    
- **【原因】**：`printf` 函数本身会占用较大的栈空间（通常几百字节），512 字节的栈极易发生**栈溢出（Stack Overflow）**。
    
- **【解决】**：将 `attr.stack_size` 放大到 **`1024`** 或 `2048` 字节，确保调试打印时的安全性。
    

## 三、 延伸收获：OpenHarmony 的启动时序认知

通过解决 `SYS_RUN(HelloWorld)` 和 `APP_FEATURE_INIT(led_demo)` 的区别，我们明确了 OpenHarmony 轻量系统的启动思想：

- **`SYS_RUN`** 属于系统早期，此时多线程调度未完全就绪，**绝对不能写 `while(1)` 死循环**，只适合做一次性的快速初始化或打印。
    
- **`APP_FEATURE_INIT`** 属于应用晚期，此时多线程环境完美就绪，**适合用来启动多任务线程**（如通过 `osThreadNew` 挂载点灯任务）。
    

这次实验帮你成功打通了从“工程配置（GN）”到“编辑器调优（VS Code）”再到“内核机制（RTOS线程与启动宏）”的闭环，是非常扎实的嵌入式开发第一步！
